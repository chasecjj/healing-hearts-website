# Design System Linting & Token Compliance — Research Plan

## Context
Stack: Tailwind CSS 3, React 19, Vite 7, GSAP animations, Scoria design system (custom tokens).
Bug patterns to catch: z-index collisions, overflow:hidden clipping, absolute-positioned SVG waves overlapping adjacent sections, hardcoded colors not matching design tokens.

## Checklist

- [x] 1. Audit current ESLint setup and identify what CSS/layout rules already exist — Clean slate: only JS recommended + React hooks/refresh. No CSS linting, no Stylelint, no Tailwind plugin.
- [x] 2. Research Stylelint + tailwind plugins for catching layout bugs (z-index, overflow, positioning) — Low ROI: project has only 2 CSS files (index.css/App.css), all styling via Tailwind utility classes in JSX. Stylelint can't lint className strings. ESLint @eslint/css is new but also CSS-file-only. stylelint-z-index-value-constraint exists for raw CSS z-index ranges but doesn't apply here.
- [x] 3. Research eslint-plugin-tailwindcss for class-level linting and token enforcement — HIGH VALUE. francoismassart/eslint-plugin-tailwindcss: 8 rules, full TW v3 support. Key rules: no-contradicting-classname (catches p-2 p-3 conflicts), no-custom-classname (restricts to TW classes + whitelist), no-unnecessary-arbitrary-value (catches m-[1.25rem] → m-5), classnames-order, enforces-shorthand. Also found: eslint-plugin-better-tailwindcss (13 rules, TW3+4, no-restricted-classes for banning specific classes). oxlint-tailwindcss has no-hardcoded-colors but is TW v4 only.
- [x] 4. Research custom Tailwind token enforcement (detecting hardcoded colors outside design tokens) — CRITICAL FINDING: 58 arbitrary hex colors across 18 files, 163 inline style objects across 22 files. #F9F8F5 alone appears 7x in Home.jsx without being a design token. Tools: eslint-plugin-tailwindcss `no-arbitrary-value` (bans all arbitrary values, too aggressive), `no-unnecessary-arbitrary-value` (catches ones with TW equivalents). oxlint-tailwindcss `no-hardcoded-colors` is ideal but TW v4 only. Best approach for TW v3: custom ESLint rule or no-restricted-classes from eslint-plugin-better-tailwindcss with regex patterns.
- [x] 5. Research visual regression / layout testing tools (Playwright visual comparisons, Storybook, etc.) — Playwright `toHaveScreenshot()` is the best fit (zero new deps, already have Playwright MCP plugin). Can catch overflow clipping, z-index overlaps, SVG wave bleeding via full-page screenshot diffs. Limitations: doesn't catch overflow:hidden clipping programmatically (known bug #33489), need custom JS assertions for bounding-box-based overlap detection. BackstopJS adds visual diff UI but 3-5 day setup cost. Recommendation: Playwright visual tests + custom layout assertions.
- [x] 6. Research GSAP-specific linting considerations (animation z-index, translate conflicts) — No GSAP-specific linting tools exist. Key issue: GSAP transforms create stacking contexts that interact with z-index and overflow:hidden. HH has 107 GSAP/ScrollTrigger calls + 89 overflow-hidden usages — high collision potential. ScrollTrigger pin:true creates pin-spacer wrappers that conflict with overflow:hidden. No static analysis can catch these; must use runtime tests (Playwright layout assertions checking bounding boxes of animated elements vs parent containers).
- [x] 7. Evaluate feasibility of each tool for the HH stack and recommend a prioritized adoption plan — See output.md for full analysis. Summary: Tier 1 (immediate, <1hr): eslint-plugin-tailwindcss with 5 recommended rules. Tier 2 (short-term, 2-4hr): custom ESLint rule for hardcoded hex detection + add #F9F8F5 etc. as design tokens. Tier 3 (medium-term): Playwright visual regression tests for overflow/z-index/wave clipping bugs. No Prettier conflict (not configured).
- [x] 8. Write findings to output.md — Complete. Full analysis with 3-tier adoption plan, code examples, and quantified codebase metrics.
